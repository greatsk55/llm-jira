# LLM-Jira

在本地环境中运行的基于LLM的项目管理工具。提供类似Jira的UI，包含利用LLM的自动化任务执行和测试功能。

## 主要功能

### 🚀 **基于域的并行执行** - 改变游戏规则的功能！
让LLM-Jira与众不同的核心特性！利用智能域标签，实现多任务无冲突同时执行。

**工作原理:**
- 为每个任务设置域标签（例如：`frontend`、`backend`、`database`、`auth`）
- 不同域的任务自动并行运行
- 相同域的任务依次等待，防止冲突
- 没有域标签？没问题 - 任务自由运行无限制

**实际使用示例:**
```
✅ 前端重设计 (frontend)    → 运行中
✅ API优化 (backend)        → 并行运行中
✅ 文档更新 (无域)          → 并行运行中
⏸️ 组件重构 (frontend)    → 等待frontend任务
```

这意味着您可以让LLM代理在处理前端改进的同时，同时处理后端优化和文档更新 - 最大化生产力，最小化空闲时间！

### 其他强大功能

- 📋 **任务管理**: 使用TODO、IN_PROGRESS、DONE、PENDING状态管理任务
- 🤖 **LLM集成**: 与Claude、ChatGPT等本地LLM CLI自动集成
- ⚡ **终端执行**: 将任务作为本地终端命令自动执行
- 📊 **实时日志**: 实时查看LLM执行内容
- 🏷️ **发布管理**: 基于Git的版本控制和快照
- 📎 **文件附件**: 可在任务中附加图片和文件
- 🔍 **搜索/过滤**: 按标题、描述、优先级搜索任务
- 🌐 **多语言支持**: UI根据浏览器语言自动切换
- 🎯 **拖放**: 通过拖放更改任务状态

## 快速开始

### 安装和运行

```bash
# 克隆仓库
git clone https://github.com/yourusername/llm-jira.git
cd llm-jira

# 安装依赖
npm install

# 初始化数据库
npm run prisma:migrate

# 启动开发服务器
npm run dev
```

服务器启动后，可以在`http://localhost:5173`访问Web UI。

### LLM CLI设置

要使用LLM集成，需要在本地安装LLM CLI:

**安装Claude CLI:**
```bash
npm install -g @anthropic-ai/claude-cli
# 或
brew install claude
```

**使用方法:**
1. 点击Web UI顶部的「LLM设置」按钮
2. 选择要使用的LLM（Claude、ChatGPT或自定义）
3. 选择首选语言
4. 保存

现在点击任务卡上的「执行LLM」按钮，任务内容和附件将自动发送给LLM。

## 使用方法

### 创建任务

在Web UI中点击「新任务」按钮创建任务。每个任务可以设置:
- 标题
- 描述
- 域标签（用于并行执行控制）
- 文件附件（可选）
- 优先级

**专业提示:** 战略性地分配域标签以最大化并行执行。例如:
- `frontend` - React组件、UI样式
- `backend` - API路由、业务逻辑
- `database` - 模式、迁移
- `tests` - 测试文件
- `docs` - 文档

### 使用LLM自动生成任务

您可以利用LLM自动将大型功能分解为带有适当域标签的可管理任务。方法如下:

**1. 任务生成提示模板:**
```
我需要实现[功能名称]。请将其分解为看板的具体、可执行任务。
对于每个任务，请提供:
- 清晰简洁的标题
- 需要完成工作的详细描述
- 适当的域标签 (frontend/backend/database/tests/docs)
- 优先级 (高/中/低)

以可直接添加到系统的格式输出任务列表。
```

**2. 示例 - 功能: "用户认证系统"**

**提示:**
```
我需要实现基于JWT的用户认证系统。请将其分解为带有域标签的任务以便并行执行。
```

**LLM输出 (并行执行结构):**
```
任务1: 设计用户数据库模式
- 域: database
- 优先级: 高
- 描述: 创建包含邮箱、密码哈希和时间戳的User模型

任务2: 创建认证API端点
- 域: backend
- 优先级: 高
- 描述: 实现使用JWT的/login、/register、/logout端点

任务3: 构建登录/注册UI组件
- 域: frontend
- 优先级: 中
- 描述: 创建用于用户认证的React表单

任务4: 编写认证测试
- 域: tests
- 优先级: 高
- 描述: 认证流程的单元测试和集成测试

任务5: 认证API文档
- 域: docs
- 优先级: 低
- 描述: 认证端点的API文档
```

**为什么这样有效:**
- ✅ 任务1、2、3、4、5都可以并行运行（不同域）
- ✅ 明确的关注点分离
- ✅ 最大化LLM代理生产力
- ✅ 减少等待依赖的空闲时间

**3. 高级: 使用LLM规划任务依赖**

对于复杂功能，让LLM识别依赖关系:

```
对于用户认证功能，哪些任务必须在其他任务之前完成？
按执行波次组织以便并行处理。
```

**示例响应:**
```
波次1 (并行执行):
- 数据库模式设计 [database]
- UI组件线框图 [frontend]
- API文档结构 [docs]

波次2 (波次1之后):
- 认证API实现 [backend]
- 登录/注册组件 [frontend]

波次3 (波次2之后):
- 集成测试 [tests]
- 最终文档 [docs]
```

**4. 最佳实践:**

✅ **建议:**
- 使用描述性的、面向行动的任务标题
- 为并行执行分配适当的域标签
- 将大任务分解为小的、可管理的部分
- 在描述中包含验收标准
- 为每个功能考虑测试任务

❌ **避免:**
- 创建过于模糊的任务（"修复错误"、"提高性能"）
- 为独立任务分配相同的域
- 忘记任务优先级
- 创建没有明确成功标准的任务

### 自动生成的规则文件

保存LLM设置后，项目根目录会自动生成指南文件(CLAUDE.md、CHATGPT.md、GEMINI.md)。这些文件包含:
- 任务完成标准
- 包括边缘情况的测试要求
- 状态管理指南
- 代码质量标准
- 安全漏洞检查

规则文件以选定的语言编写，帮助LLM按高质量标准执行任务。
