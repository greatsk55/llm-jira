# LLM-Jira

로컬 환경에서 실행되는 LLM 기반 프로젝트 관리 도구입니다. Jira와 유사한 UI를 제공하며, LLM을 활용한 자동화된 작업 실행 및 테스트 기능을 포함합니다.

## 주요 기능

### 🚀 **도메인 기반 병렬 실행** - 최고의 차별화 포인트!
LLM-Jira를 특별하게 만드는 핵심 기능! 지능적인 도메인 태깅을 활용하여 충돌 없이 여러 작업을 동시에 실행할 수 있습니다.

**작동 원리:**
- 각 작업에 도메인 태그 지정 (예: `frontend`, `backend`, `database`, `auth`)
- 다른 도메인의 작업들은 자동으로 병렬 실행
- 같은 도메인의 작업들은 차례를 기다려 충돌 방지
- 도메인 없는 작업은 제한 없이 자유롭게 실행

**실제 사용 예시:**
```
✅ 프론트엔드 리디자인 (frontend)  → 실행 중
✅ API 최적화 (backend)           → 병렬 실행 중
✅ 문서 업데이트 (도메인 없음)     → 병렬 실행 중
⏸️ 컴포넌트 리팩토링 (frontend)  → frontend 작업 대기 중
```

이것은 LLM 에이전트가 프론트엔드 개선 작업을 하는 동안 동시에 백엔드 최적화와 문서 업데이트를 처리할 수 있다는 의미입니다 - 생산성을 극대화하고 유휴 시간을 최소화합니다!

- ⚡ **터미널 실행**: 작업을 로컬 터미널 명령으로 자동 실행

- 🔍 **검색/필터**: 제목, 설명, 우선순위로 작업 검색

## 빠른 시작

### 설치 및 실행

```bash
# 저장소 클론
git clone https://github.com/yourusername/llm-jira.git
cd llm-jira

# 의존성 설치
npm install

# 데이터베이스 초기화
npm run prisma:migrate

# 개발 서버 시작
npm run dev
```

서버가 시작되면 `http://localhost:5173`에서 웹 UI에 접속할 수 있습니다.

### LLM CLI 설정

LLM 통합을 사용하려면 로컬에 LLM CLI를 설치해야 합니다:

**Claude CLI 설치:**
```bash
npm install -g @anthropic-ai/claude-cli
# 또는
brew install claude
```

**사용법:**
1. 웹 UI 상단의 "LLM 설정" 버튼 클릭
2. 사용할 LLM 선택 (Claude, ChatGPT, 또는 커스텀)
3. 선호하는 언어 선택
4. 저장

이제 작업 카드에서 "LLM 실행" 버튼을 클릭하면 작업 내용과 첨부 파일이 자동으로 LLM에게 전달됩니다.

## 사용법

### 작업 생성

웹 UI에서 "새 작업" 버튼을 클릭하여 작업을 생성합니다. 각 작업에는:
- 제목
- 세부 내용
- 도메인 태그 (병렬 실행 제어용)
- 첨부 파일 (선택사항)
- 우선순위

를 설정할 수 있습니다.

**팁:** 병렬 실행을 최대화하려면 도메인 태그를 전략적으로 할당하세요. 예를 들어:
- `frontend` - React 컴포넌트, UI 스타일링
- `backend` - API 라우트, 비즈니스 로직
- `database` - 스키마, 마이그레이션
- `tests` - 테스트 파일
- `docs` - 문서

### LLM을 활용한 자동 작업 생성

LLM을 활용하여 대규모 기능을 적절한 도메인 태그가 포함된 관리 가능한 작업으로 자동 분해할 수 있습니다. 방법은 다음과 같습니다:

**1. 작업 생성용 프롬프트 템플릿:**
```
[기능명]을 구현해야 합니다. 이를 칸반 보드용으로 구체적이고 실행 가능한 작업들로
나눠주세요. 각 작업마다 다음을 제공해주세요:
- 명확하고 간결한 제목
- 수행해야 할 작업의 상세 설명
- 적절한 도메인 태그 (frontend/backend/database/tests/docs)
- 우선순위 (높음/보통/낮음)

시스템에 바로 추가할 수 있는 형식으로 작업 목록을 출력해주세요.
```

**2. 예시 - 기능: "사용자 인증 시스템"**

**프롬프트:**
```
JWT 기반 사용자 인증 시스템을 구현해야 합니다. 병렬 실행을 위한 도메인 태그와 함께
작업으로 나눠주세요.
```

**LLM 출력 (병렬 실행 구조):**
```
작업 1: 사용자용 데이터베이스 스키마 설계
- 도메인: database
- 우선순위: 높음
- 설명: 이메일, 비밀번호 해시, 타임스탬프가 포함된 User 모델 생성

작업 2: 인증 API 엔드포인트 생성
- 도메인: backend
- 우선순위: 높음
- 설명: JWT를 사용하는 /login, /register, /logout 엔드포인트 구현

작업 3: 로그인/회원가입 UI 컴포넌트 구축
- 도메인: frontend
- 우선순위: 보통
- 설명: 사용자 인증용 React 폼 생성

작업 4: 인증 테스트 작성
- 도메인: tests
- 우선순위: 높음
- 설명: 인증 플로우를 위한 단위 및 통합 테스트

작업 5: 인증 API 문서화
- 도메인: docs
- 우선순위: 낮음
- 설명: 인증 엔드포인트용 API 문서
```

**왜 이것이 효과적인가:**
- ✅ 작업 1, 2, 3, 4, 5가 모두 병렬 실행 가능 (서로 다른 도메인)
- ✅ 명확한 관심사 분리
- ✅ LLM 에이전트 생산성 극대화
- ✅ 의존성 대기 유휴 시간 감소

**3. 고급: LLM을 사용한 작업 의존성 계획**

복잡한 기능의 경우 LLM에게 의존성을 파악하도록 요청하세요:

```
사용자 인증 기능에서 어떤 작업이 다른 작업보다 먼저 완료되어야 하나요?
병렬 처리를 위해 실행 웨이브별로 정리해주세요.
```

**예시 응답:**
```
웨이브 1 (병렬 실행):
- 데이터베이스 스키마 설계 [database]
- UI 컴포넌트 와이어프레임 [frontend]
- API 문서 구조 [docs]

웨이브 2 (웨이브 1 이후):
- 인증 API 구현 [backend]
- 로그인/회원가입 컴포넌트 [frontend]

웨이브 3 (웨이브 2 이후):
- 통합 테스트 [tests]
- 최종 문서화 [docs]
```

**4. 모범 사례:**

✅ **권장사항:**
- 설명적이고 행동 지향적인 작업 제목 사용
- 병렬 실행을 위한 적절한 도메인 태그 할당
- 큰 작업을 작고 관리 가능한 조각으로 분해
- 설명에 완료 기준 포함
- 각 기능에 대한 테스트 작업 고려

❌ **피해야 할 것:**
- 너무 모호한 작업 생성 ("버그 수정", "성능 개선")
- 독립적인 작업에 같은 도메인 할당
- 작업 우선순위 지정 누락
- 명확한 성공 기준 없는 작업 생성

### 작업 실행 (터미널 명령)

API를 통해 작업을 실행할 수 있습니다:

```bash
# 작업 실행
curl -X POST http://localhost:3000/api/tasks/{issueId}/execute \
  -H "Content-Type: application/json" \
  -d '{"command": "npm test", "llmProvider": "claude"}'

# 현재 실행 중인 도메인 확인 (병렬 실행용)
curl http://localhost:3000/api/tasks/running

# 실행 상태 확인
curl http://localhost:3000/api/tasks/{issueId}/status

# 실행 취소
curl -X POST http://localhost:3000/api/tasks/{issueId}/cancel
```

작업 실행 시:
1. 시스템이 도메인 충돌 확인 - 같은 도메인의 작업은 동시 실행 불가
2. 상태가 자동으로 IN_PROGRESS로 변경됩니다
3. 터미널에서 지정된 명령이 백그라운드로 실행됩니다
4. stdout/stderr 출력이 기록됩니다
5. 성공 시 상태가 DONE으로, 실패 시 PENDING으로 변경됩니다
6. 실행 기록이 Execution 테이블에 저장됩니다
7. 도메인이 같은 도메인의 다른 작업을 위해 해제됩니다

**도메인 충돌 처리:**
- 작업의 도메인이 이미 실행 중이면 API가 409 (충돌) 반환
- 도메인 태그가 없는 작업은 절대 충돌하지 않음
- 다른 도메인의 여러 작업이 원활하게 병렬 실행됨

### 릴리즈 생성

"릴리즈" 메뉴에서 릴리즈를 생성할 수 있습니다:
- 버전 번호 입력 (예: v1.0.0)
- 현재 작업 상태가 스냅샷으로 저장됩니다
- Git 커밋 및 태그가 자동으로 생성됩니다

### 버전 체크아웃

릴리즈 목록에서 특정 버전을 선택하여 해당 버전의 작업 상태로 복원할 수 있습니다.

## LLM 설정

### 자동 생성되는 룰 파일

LLM 설정을 저장하면 프로젝트 루트에 가이드라인 파일(CLAUDE.md, CHATGPT.md, GEMINI.md)이 자동으로 생성됩니다. 이 파일들은 다음을 포함합니다:
- 작업 완료 기준
- 엣지 케이스를 포함한 테스트 요구사항
- 상태 관리 가이드라인
- 코드 품질 기준
- 보안 취약점 체크

룰 파일은 선택한 언어로 작성되며 LLM이 높은 품질 기준으로 작업을 실행하도록 돕습니다.
