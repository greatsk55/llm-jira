# LLM-Jira

ローカル環境で実行されるLLMベースのプロジェクト管理ツールです。JiraのようなUIを提供し、LLMを活用した自動化されたタスク実行とテスト機能を含みます。

## 主な機能

### 🚀 **ドメインベース並列実行** - ゲームチェンジャー!
LLM-Jiraを特別なものにする最大の特徴！インテリジェントなドメインタグを活用して、競合なしで複数のタスクを同時実行できます。

**仕組み:**
- 各タスクにドメインタグを設定（例: `frontend`、`backend`、`database`、`auth`）
- 異なるドメインのタスクは自動的に並列実行
- 同じドメインのタスクは順番を待って競合を防止
- ドメインなし？問題なし - 制限なしで自由に実行

**実際の使用例:**
```
✅ フロントエンド再設計 (frontend)  → 実行中
✅ API最適化 (backend)              → 並列実行中
✅ ドキュメント更新 (ドメインなし)  → 並列実行中
⏸️ コンポーネントリファクタ (frontend) → frontend待機中
```

これは、LLMエージェントがフロントエンド改善作業をしながら、同時にバックエンド最適化とドキュメント更新を処理できることを意味します - 生産性を最大化し、アイドル時間を最小化！

### その他の強力な機能

- 📋 **タスク管理**: TODO、IN_PROGRESS、DONE、PENDINGステータスでタスクを管理
- 🤖 **LLM統合**: ClaudeやChatGPTなどのローカルLLM CLIと自動連携
- ⚡ **ターミナル実行**: タスクをローカルターミナルコマンドとして自動実行
- 📊 **リアルタイムログ**: LLM実行内容をリアルタイムで確認
- 🏷️ **リリース管理**: Gitベースのバージョン管理とスナップショット
- 📎 **ファイル添付**: タスクに画像やファイルを添付可能
- 🔍 **検索/フィルター**: タイトル、説明、優先度でタスクを検索
- 🌐 **多言語対応**: ブラウザ言語に応じてUIが自動切り替え
- 🎯 **ドラッグ&ドロップ**: ドラッグ&ドロップでタスクのステータスを変更

## クイックスタート

### インストールと実行

```bash
# リポジトリをクローン
git clone https://github.com/yourusername/llm-jira.git
cd llm-jira

# 依存関係をインストール
npm install

# データベースを初期化
npm run prisma:migrate

# 開発サーバーを起動
npm run dev
```

サーバーが起動したら、`http://localhost:5173`でWeb UIにアクセスできます。

### LLM CLI セットアップ

LLM統合を使用するには、ローカルにLLM CLIをインストールする必要があります:

**Claude CLIのインストール:**
```bash
npm install -g @anthropic-ai/claude-cli
# または
brew install claude
```

**使い方:**
1. Web UI上部の「LLM設定」ボタンをクリック
2. 使用するLLMを選択（Claude、ChatGPT、またはカスタム）
3. 好みの言語を選択
4. 保存

これでタスクカードの「LLM実行」ボタンをクリックすると、タスクの内容と添付ファイルが自動的にLLMに送信されます。

## 使い方

### タスク作成

Web UIで「新規タスク」ボタンをクリックしてタスクを作成します。各タスクには以下を設定できます:
- タイトル
- 説明
- ドメインタグ（並列実行制御用）
- ファイル添付（オプション）
- 優先度

**プロのヒント:** 並列実行を最大化するために、戦略的にドメインタグを割り当てましょう。例えば:
- `frontend` - Reactコンポーネント、UIスタイリング
- `backend` - APIルート、ビジネスロジック
- `database` - スキーマ、マイグレーション
- `tests` - テストファイル
- `docs` - ドキュメント

### LLMを活用した自動タスク生成

LLMを活用して、大規模な機能を適切なドメインタグ付きの管理可能なタスクに自動分解できます。方法は以下の通りです:

**1. タスク生成用プロンプトテンプレート:**
```
[機能名]を実装する必要があります。これをカンバンボード用の具体的で実行可能な
タスクに分解してください。各タスクについて以下を提供してください:
- 明確で簡潔なタイトル
- 実行すべき作業の詳細な説明
- 適切なドメインタグ (frontend/backend/database/tests/docs)
- 優先度レベル (高/中/低)

システムに直接追加できる形式でタスクリストを出力してください。
```

**2. 例 - 機能: "ユーザー認証システム"**

**プロンプト:**
```
JWTベースのユーザー認証システムを実装する必要があります。並列実行のための
ドメインタグと共にタスクに分解してください。
```

**LLM出力 (並列実行構造):**
```
タスク1: ユーザー用データベーススキーマ設計
- ドメイン: database
- 優先度: 高
- 説明: メール、パスワードハッシュ、タイムスタンプを含むUserモデル作成

タスク2: 認証APIエンドポイント作成
- ドメイン: backend
- 優先度: 高
- 説明: JWTを使用した/login、/register、/logoutエンドポイント実装

タスク3: ログイン/登録UIコンポーネント構築
- ドメイン: frontend
- 優先度: 中
- 説明: ユーザー認証用Reactフォーム作成

タスク4: 認証テスト作成
- ドメイン: tests
- 優先度: 高
- 説明: 認証フローの単体テストと統合テスト

タスク5: 認証API文書化
- ドメイン: docs
- 優先度: 低
- 説明: 認証エンドポイントのAPIドキュメント
```

**なぜこれが効果的か:**
- ✅ タスク1、2、3、4、5がすべて並列実行可能（異なるドメイン）
- ✅ 明確な関心の分離
- ✅ LLMエージェントの生産性を最大化
- ✅ 依存関係待ちのアイドル時間を削減

**3. 高度: LLMを使用したタスク依存関係の計画**

複雑な機能の場合、LLMに依存関係を特定させましょう:

```
ユーザー認証機能で、どのタスクが他のタスクより先に完了する必要がありますか?
並列処理のために実行ウェーブごとに整理してください。
```

**例の応答:**
```
ウェーブ1 (並列実行):
- データベーススキーマ設計 [database]
- UIコンポーネントワイヤーフレーム [frontend]
- APIドキュメント構造 [docs]

ウェーブ2 (ウェーブ1の後):
- 認証API実装 [backend]
- ログイン/登録コンポーネント [frontend]

ウェーブ3 (ウェーブ2の後):
- 統合テスト [tests]
- 最終ドキュメント化 [docs]
```

**4. ベストプラクティス:**

✅ **推奨:**
- 説明的でアクション指向のタスクタイトルを使用
- 並列実行のための適切なドメインタグを割り当て
- 大きなタスクを小さく管理可能な部分に分解
- 説明に完了基準を含める
- 各機能のテストタスクを考慮

❌ **避けるべきこと:**
- 曖昧すぎるタスクの作成（「バグ修正」、「パフォーマンス改善」）
- 独立したタスクに同じドメインを割り当て
- タスクの優先順位付けを忘れる
- 明確な成功基準のないタスクを作成

### 自動生成されるルールファイル

LLM設定を保存すると、プロジェクトルートにガイドラインファイル(CLAUDE.md、CHATGPT.md、GEMINI.md)が自動生成されます。これらのファイルには以下が含まれます:
- タスク完了基準
- エッジケースを含むテスト要件
- ステータス管理ガイドライン
- コード品質基準
- セキュリティ脆弱性チェック

ルールファイルは選択した言語で記述され、LLMが高品質基準でタスクを実行するのを助けます。
